# Impeccable — ADB Auth

Design context for AI-assisted UI work, aligned with maintainer intent and the current codebase.

## Design Context

### Users

**Who:** **Beginners and experts** who use **wireless ADB**—first-time wireless debug users and experienced Android developers.

**Context:** On a **desktop** (Electron), often alongside an IDE or terminal, with a phone on the same network. They need to **pair** (QR or pairing code), **connect** to the debug endpoint, **see device state**, and **fix ADB path / refresh** when something breaks.

**Job to be done:** Complete pairing and connection with **minimal friction**, understand status at a glance, and trust that tooling stays **local** (no cloud, no telemetry—per README).

**UX implication for mixed skill levels:** Favor **clear defaults and guided steps** for newcomers without hiding **speed and density** for experts (short paths, keyboard-friendly flows where natural, no unnecessary wizardry).

### Brand Personality

**Three words:** **Fast**, **easy**, **clean.**

**Supporting traits (from product):** **Trustworthy** and **local-first**—clear labels, security-minded README, no marketing fluff.

**Emotional target:** Confidence and low friction: the app feels **quick to use**, **simple to understand**, and **visually uncluttered**; errors stay **actionable** so neither beginners nor experts get stuck guessing.

### Aesthetic Direction

**Theme:** **Dark only**—no light mode planned; keep improving the existing dark glass system rather than introducing a second theme unless requirements change.

**Visual tone:** **Deep navy** base (`--bg-base`), **frosted glass** surfaces (`glass-panel`, `glass-panel-strong`, backdrop blur), **soft blue/green** ambient glows (decorative layers in `App.tsx`).

**Typography:** **IBM Plex Sans** for UI; **JetBrains Mono** for codes, addresses, and technical strings (`.font-code`).

**Color roles:** **Green** as primary/success accent (`--accent-primary`, primary buttons); **blue** for links, focus rings, and active nav emphasis; **red/amber** for danger and warning states as defined in CSS variables.

**Components:** Tailwind CSS v4 + shared utility classes in `src/index.css` (`.ui-btn*`, `.ui-input`, `.ui-banner*`, `.ui-chip`, `.ui-label`). **Lucide React** for icons. Layout: **sidebar + main glass panel** + **status bar**.

**“Clean” in practice:** Prefer **clear hierarchy**, **consistent spacing**, and **restraint** on decoration—enough polish to feel modern, not enough noise to slow scanning.

### Design Principles

1. **Fast, easy, clean** — Optimize for **speed** (few steps, obvious primary actions), **ease** (plain language, sensible defaults for beginners), and **visual clarity** (no clutter, strong hierarchy).
2. **Beginner–expert balance** — Don’t assume prior ADB lore in copy; keep **advanced details** available where experts expect them (ports, paths, status) without blocking simple flows.
3. **Local trust** — UI and copy reinforce **privacy** and **local-only** operation; never imply cloud sync or tracking.
4. **Token-first styling** — Prefer `:root` CSS variables and existing `.ui-*` / `.glass-*` patterns before one-off hex values; keep text readable on dark glass (`--text-primary` → `--text-faint`).
5. **State visibility** — Pairing progress, device online/offline, and ADB path should be obvious; errors explain **what to try next** (align with README troubleshooting).
6. **Accessible for all** — Treat accessibility as a **first-class requirement**: semantic structure, names/labels for controls, visible focus (`--focus-ring`), skip link to main content, `prefers-reduced-motion`, sufficient **contrast** on dark surfaces, and **target sizes** that work for pointer and keyboard users. Aim for **WCAG 2.1 Level AA** for new and revised UI unless a specific pattern is technically blocked—then document the exception and mitigate.
